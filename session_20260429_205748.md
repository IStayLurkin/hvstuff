## BSOD Analysis (run 5, 20:43)

```
DRIVER_IRQL_NOT_LESS_OR_EQUAL (0xD1)
  P1 = 0x2b   (address referenced)
  P2 = 0xFF   (IRQL — KPRCB corruption)
  P4 = 0x2b   (faulting instruction address)

Faulting IP = 0x2b = 64-bit user-mode CS selector on Windows x64
```

`AsmVmExitHandler` executed `jmp r14` where `r14 = 0x2b`. That value came from reading `[HOST_RSP+8]` (the pool buffer slot). The slot should have contained the `launch_resume` label address written by `AsmLaunchAndReturn` — but instead held `0x2b`, causing a jump into near-zero memory → page fault at elevated IRQL → KPRCB corruption → BSOD.

## Root Cause

The **host-stack-slot scheme** was unreliable. `AsmLaunchAndReturn` wrote the resume RIP to `[r15+8]` (where `r15 = hostRsp`, the pool buffer midpoint). `AsmVmExitHandler` read it back from `[rsp+8]` (where `rsp = HOST_RSP` on VM-exit). These should have been the same address, but `0x2b` ended up at that location — either due to an undetected addressing discrepancy or pool memory being partially re-initialized.

## Fix Applied

Replaced the stack-slot scheme with three **non-paged C globals** in `Vmx.c`:

```c
ULONG64 g_HostResumeRip = 0;
ULONG64 g_HostResumeRsp = 0;
ULONG   g_ExitReason    = 0xFFFFFFFF;
```

Declared via `EXTERNDEF` in `Arch.asm`. Changes:

| File | Change |
|------|--------|
| `Vmx.c` | Added three globals; post-launch reads `g_ExitReason` instead of `*(ULONG*)(hostRsp+24)` |
| `Vmx.h` | Added `extern` declarations for the three globals |
| `Arch.asm` | Added `EXTERNDEF` block; `AsmLaunchAndReturn` writes globals before vmlaunch; `AsmVmExitHandler` reads globals instead of `[rsp+8]`/`[rsp+16]` |

Disassembly confirmed correct encoding:
- `mov qword ptr [g_HostResumeRip], rax` — RIP-relative store
- `mov r14, qword ptr [g_HostResumeRip]` — RIP-relative load
- `vmread rax, rcx` → `0F 78 C8` ✓
- `jmp r14` → `41 FF E6` ✓

HOST_RSP (pool midpoint) is still written to VMCS so the CPU has a valid stack on VM-exit, but no slot values are stored there anymore.
